## 摘要

本报告针对无线基站团队提出的"砍掉研发90%人力投资"目标，系统调研了Harness Engineering在业界的实践，并基于1亿行C/C++嵌入式代码的具体场景设计了完整落地方案。

核心结论：砍掉90%人力的目标在技术上是可行的，但需要分阶段实施，初期目标应设定为**砍掉60-70%**，通过3-5年过渡期逐步实现。完全无人化不现实，但可以通过Harness Engineering将人力从重复性工作中解放出来，转向更高价值的设计创新和AI系统优化工作。

## 第一部分：调研分析

### 1.1 Harness Engineering 定义与内涵

Harness Engineering（测试 harness/开发 harness/CI/CD harness/开发环境 harness）并非新概念，但在AI Agent时代被赋予了全新内涵。

**传统定义**：
- Test Harness：测试驱动框架，提供测试执行环境、桩函数Mock、数据注入
- Build Harness：自动化构建环境，编译脚本、依赖管理、构建流水线
- CI/CD Harness：持续集成/部署管道，门禁检查、自动化验证、部署策略

**AI时代的Harness Engineering**：

2025年OpenAI发布的[Harness Engineering实践](https://openai.com/index/harness-engineering/)重新定义了这一概念：

> 过去五个月，我们的团队做了一个实验：用0行手动编写的代码构建并发布了一个软件产品。产品有内部日活用户和外部alpha测试者，能上线、部署、故障和修复。不同之处在于，每一行代码——应用逻辑、测试、CI配置、文档、可观测性、内部工具——都是由Codex编写的。我们估计这花了大约十分之一的手写代码时间。

关键数据：
- **1,000,000+行代码**（无一行人工手写）
- **5个月**完成从零到产品上线
- **1/10时间**（相比人工开发）
- **3.5 PRs/工程师/天**的平均吞吐量
- **7人团队**驱动整个系统

### 1.2 Harness Engineering 三大支柱

OpenAI团队将Harness Engineering分解为三个核心领域：

#### 1.2.1 Context Engineering（上下文工程）

**核心原则**：给Agent一张地图，而不是1000页说明书

- **结构化知识库**：docs/目录作为系统级知识存储，而非巨型AGENTS.md
- **渐进式披露**：Agent从小型稳定入口开始，需要时再深入
- **机械验证**：专门的linters和CI jobs验证知识库是最新的、正确交叉链接的
- **Doc-gardening Agent**：定期扫描并修复过时的文档

**实践要点**：
- 知识必须存在于代码仓库内部（代码、markdown、schema、可执行计划）
- 外部文档（Google Docs、Slack讨论）对Agent不可见 = 对新员工也不可见
- 设计文档需编入索引，包含验证状态和核心信念

#### 1.2.2 Architectural Constraints（架构约束）

**核心原则**：Agent在最严格的环境中最高效

- **边界解析**：在边界处必须解析数据形状，而非仅验证类型（"Parse, don't validate"原则）
- **自定义Linters**：非仅依赖通用工具，需要针对项目特定约束
- **结构化测试**：如ArchUnit的架构测试思想
- **入口点控制**：Agent只能访问其任务所需的上下文

**关键洞察**：
- 看似"无聊"的技术栈（boring technologies）反而更适合AI，因为：
  - 可组合性好
  - API稳定性高
  - 训练数据充分
- 有时让Agent重新实现子功能比适配不透明的库更便宜

#### 1.2.3 Garbage Collection（垃圾回收）

**核心原则**：Agent必须对抗熵增

- **定期Agent**：周期性运行以发现：
  - 文档不一致
  - 架构约束违反
  - 技术债务累积
- **自我修复循环**：Agent失败时，人类工程师介入询问"缺少什么能力？如何让Agent可理解可执行？"
- **Agent-to-Agent Review**：代码审查也逐步转向AI-Agent之间的循环

**组织级洞察**：

> 我们最困难的挑战现在集中在设计环境、反馈循环和控制系统。  
> —— OpenAI Codex团队

### 1.5 "砍90%人力"可行性分析

| 环节 | 替代可能性 | 备注 |
|------|-----------|------|

#### 1.5.2 难以完全替代环节（需要人）

| 环节 | 原因 | 建议保留比例 |
|------|------|--------------|
| 架构设计 | 需要业务理解、硬件知识、系统级思考 | 80-100% |
| 协议栈设计 | 3GPP标准演进、硬件兼容性 | 100% |
| 硬件抽象层 | 硬件相关性高，需要硬件团队协同 | 100% |
| 新技术预研 | 探索性工作，AI难以胜任 | 100% |
| 复杂问题诊断 | 需要领域知识 + 硬件调试能力 | 60-80% |
| 安全/合规审查 | 法规要求、认证需求 | 100% |
| 客户定制支持 | 业务差异化需求 | 视情况 |

#### 1.5.3 可行性结论

**短期（1-2年）目标**：
- 砍掉40-50%重复性工作人力
- 转向：Harness建设、AI调优、架构优化

**中期（3-5年）目标**：
- 砍掉60-70%人力
- 核心架构/硬件团队保留（15-20%）
- AI运维/监控系统团队（10-15%）

**长期（5年+）目标**：
- 砍掉70-80%人力
- 剩余：架构师 + 硬件专家 + AI系统优化师 + 安全合规

**注意**：90%目标在嵌入式基站领域**不现实**，因为：
1. 硬件强相关代码无法AI化
2. 电信行业安全合规要求必须人工审查
3. 现场问题需要人类专家诊断

### 2.2 代码库治理方案

#### 2.2.1 Monorepo vs Multi-Repo

**推荐：混合策略**

```
┌─────────────────────────────────────────────────────────┐
│                    顶层：单一Monorepo                      │
│   (统一版本、共享依赖、原子提交)                          │
├─────────────────────────────────────────────────────────┤
│  L1: 物理/驱动层  │  独立仓库  │  硬件强耦合             │
│  L2: 物理层抽象   │  Monorepo  │  每年更新              │
│  L3: 协议栈      │  Monorepo  │  核心开发区            │
│  L4: 应用层      │  Monorepo  │  快速迭代               │
└─────────────────────────────────────────────────────────┘
```

**理由**：
- 驱动层：硬件相关，独立版本周期
- 协议栈：核心资产，统一维护
- 应用层：快速迭代，灵活部署

#### 2.2.2 模块化与接口契约

**核心原则**：
1. **每个模块有且只有一个明确定义的责任**
2. **模块间通过Versioned IDL通信**
3. **模块边界有Architecture Tests守护**

```
// 示例：模块接口契约 (使用Protocol Buffers)
syntax = "proto3";

package基站.v1;

// 物理层抽象接口
service PhysicalLayer {
  // 唯一入口：配置
  rpc Configure(PhysicalLayerConfig) returns (ConfigureResult);
  
  // 唯一入口：数据传输
  rpc Transmit(TransmitRequest) returns (TransmitResponse);
  
  // 唯一入口：状态查询
  rpc GetStatus(Empty) returns (PhysicalLayerStatus);
}

// 版本化契约
message PhysicalLayerConfig {
  option (versioned) = "1.0.0";
  
  string device_id = 1;
  int32 carrier_id = 2;
  TransmissionParams tx_params = 3;
  
  // 契约兼容性规则
  reserved 4;  // 旧字段必须标记reserved
}
```

## 第三部分：风险评估与缓解

### 3.1 主要风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| AI生成代码质量不稳定 | 高 | 高 | 渐进式采用 + 严格门禁 |
| 团队技能转型困难 | 中 | 高 | 内部培训 + 外部招聘 |
| 供应商锁定 | 中 | 中 | 开源工具为主 |
| 安全合规审查无法自动化 | 高 | 高 | 保留人工审查 + 自动化辅助 |
| 硬件迭代周期长 | 中 | 中 | 软硬分离开发 |

### 3.2 关键成功因素

1. **高层支持**：必须有一致的技术愿景
2. **渐进式采用**：不追求一步到位
3. **文化建设**：拥抱失败，从失败中学习
4. **持续投资**：Harness是长期工程
5. **人才保留**：核心人员不可流失
