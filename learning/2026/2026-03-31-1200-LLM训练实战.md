> 学习虾 · David 硬核学习计划 · 2026-03-31 12:00
> 目标：理解 ZeRO 分片策略，能根据显存选择合适的并行方案

## 一、为什么显存永远不够用？

### 1.1 一个数字说清楚问题

以 GPT-3 175B 为例：

```
每个参数：FP16（半精度） = 2 bytes
175B 参数 × 2 bytes = 350 GB

单卡最强显存（A100/H100） = 80 GB

缺口：350 ÷ 80 = 4.4 倍
```

这只是**放进去**的体积。训练比推理更吃显存：

| 显存消耗项 | 公式 | 175B 模型实际占用 |
|:---|:---|:---|
| 模型参数（FP16） | 参数量 × 2B | 350 GB |
| 梯度 | 参数量 × 2B | 350 GB |
| 优化器状态（Adam） | 参数量 × 4B（FP32 m+v） | 700 GB |
| 激活值 | BatchSize × SeqLen × 层数 × 隐藏维度 | ~100 GB（视配置） |
| **训练总计** | | **~1500 GB** |

1500 GB，需要 **19 张 80GB A100** 才能装下。这还没算 KV Cache 和训练时的中间变量。

这就是显存墙（Memory Wall）。不解决显存问题，模型再大也训不动。

### 1.2 两类解决方案的思路

**方案 A：改变硬件互联方式** — 让多张卡像一个整体一样工作（张量并行 TP、流水线并行 PP）

**方案 B：减少每张卡存的内容** — 把完整状态分片，分散到多张卡上（ZeRO）

ZeRO 属于方案 B，核心思想：**不改变模型结构，只改变存储方式**。

## 二、数据并行 vs 模型并行：根本区别在哪？

很多人混淆这两个概念，先说清楚。

### 2.1 数据并行（DP）：模型完全复制，每个 GPU 只拿不同的数据

```
场景：4 张 GPU，模型参数 350GB

每张卡都有一份完整模型副本：
GPU 0: [完整模型 350GB] + [数据 batch A]
GPU 1: [完整模型 350GB] + [数据 batch B]
GPU 2: [完整模型 350GB] + [数据 batch C]
GPU 3: [完整模型 350GB] + [数据 batch D]

→ 每张卡各自独立前向 + 反向
→ AllReduce 汇总梯度（所有卡拿到相同的平均梯度）
→ 每张卡独立更新相同模型
```

**本质**：GPU 之间模型状态**完全相同**，只是处理的数据不同。

**关键问题**：每张卡都要装下完整模型，**显存一个字节都没省**。吞吐提升了，但显存压力不变。

### 2.2 模型并行（TP/PP）：每个 GPU 只有模型的一部分，数据完全相同

```
张量并行（TP）例子：把一个矩阵乘法拆分到 4 张卡
原始：Y = X · W，W [4096, 14336]
拆分后每张卡只存 W 的 1/4 列：
GPU 0: W₀ [4096, 3584]，处理相同的 X，输出 Y₀
GPU 1: W₁ [4096, 3584]，处理相同的 X，输出 Y₁
... → 4 卡输出拼接成完整 Y

流水线并行（PP）例子：把 40 层拆分到 4 张卡
GPU 0: 层 0-9
GPU 1: 层 10-19
GPU 2: 层 20-29
GPU 3: 层 30-39
所有 GPU 处理相同数据，只是各自负责不同的层
```

**本质**：每张卡只持有模型的一个**物理分片**，相互依赖才能完成计算。

### 2.3 两者的根本区别

| 维度 | 数据并行 | 模型并行（TP/PP） |
|:---|:---|:---|
| **模型状态** | 每卡完整副本 | 每卡只有一部分 |
| **显存节省** | ❌ 不省 | ✅ 显著节省 |
| **通信内容** | 梯度（每步一次 AllReduce）| 激活值（多次 AllGather/P2P）|
| **通信模式** | 所有卡两两通信 | 卡间需要按顺序传递 |
| **计算依赖** | 无依赖，可完全并行 | 有依赖，必须等待 |
| **适用场景** | 模型能装进单卡 | 模型装不进单卡 |

**数据并行解决的是"训练快不快"，模型并行解决的是"模型能不能跑起来"。**

ZeRO 本质上是在数据并行的框架下，用分片技术减少每张卡的显存占用——它保留了 DP 的简单通信模式（AllReduce），同时让每张卡不用存完整状态。

## 三、ZeRO：三阶段分片策略

ZeRO（Zero Redundancy Optimizer）来自微软 DeepSpeed，核心思路：**每个 GPU 只存 1/N 的训练状态，不存冗余副本**。

N = GPU 数量。

### 3.1 问题：训练状态里有什么冗余？

普通 DataParallel 下，每个 GPU 都存了完整的三类状态：
- 优化器状态（Adam m/v）：每个参数对应 2 个 FP32 → 700 GB（175B）
- 梯度：FP16 → 350 GB（175B）
- 参数本身：FP16 → 350 GB（175B）

但实际上，每张卡只需要自己的 1/N 份。**其余 N-1 份是冗余的**，ZeRO 要把这些冗余全部消掉。

### 3.2 ZeRO Stage 1：只分片优化器状态

每张卡只存 1/N 的优化器状态（Adam m/v）。

```
普通 DP：每卡存 [全部参数副本] + [全部梯度副本] + [全部 Adam 状态副本]
ZeRO-1：每卡存 [全部参数副本] + [全部梯度副本] + [1/N Adam 状态]

175B 模型，N=8：
节省：700GB × (1 - 1/8) ≈ 613 GB（主要省的是 Adam 状态）
每卡显存：从 ~1400GB → ~787GB
```

**通信代价**：几乎为零。优化器状态只在参数更新时需要，AllReduce 通信模式不变。

### 3.3 ZeRO Stage 2：分片优化器状态 + 梯度

每张卡只存 1/N 的优化器状态 + 1/N 的梯度。

```
ZeRO-2：每卡存 [全部参数副本] + [1/N 梯度] + [1/N Adam 状态]

175B 模型，N=8：
节省：700GB × (1 - 1/8) + 350GB × (1 - 1/8) ≈ 919 GB
每卡显存：从 ~1400GB → ~481GB
```

**通信代价**：梯度 AllReduce 需要改为分片版本的 reduce-scatter + allgather，通信量约增加 1.5 倍，但完全可以接受。

### 3.4 ZeRO Stage 3：分片所有状态（参数 + 梯度 + 优化器）

每张卡只存 1/N 的完整模型状态。

```
ZeRO-3：每卡存 [1/N 参数] + [1/N 梯度] + [1/N Adam 状态]

175B 模型，N=8：
每类状态都分片到 8 张卡
每卡显存：1400GB ÷ 8 ≈ 175 GB（理论上）
```

175GB 已经可以在单张 A100（80GB）× 3 的配置下跑起来了（配合激活值压缩）。

**通信代价**：最高。前向传播时需要按需 AllGather 参数（每次层计算前），反向传播时需要类似操作。通信量约为普通 DP 的 3 倍。

### 3.5 三阶段对比总结

| 维度 | ZeRO-1 | ZeRO-2 | ZeRO-3 |
|:---|:---|:---|:---|
| **分片内容** | 优化器状态 | 优化器状态 + 梯度 | 参数 + 梯度 + 优化器 |
| **显存节省（N=8）** | ~4 倍 | ~8 倍 | ~8 倍 |
| **通信增量** | ~0 | ~1.5× | ~3× |
| **实现复杂度** | 低 | 中 | 高 |
| **适用规模** | 几亿 ~ 几十亿参数 | 几十亿 ~ 几百亿参数 | 百亿参数以上 |
| **典型场景** | 单节点多卡 | 多节点 | 超大规模训练 |

**显存节省计算公式（ZeRO-3）**：

```
每卡显存 ≈ (模型参数 × 2 + 梯度 × 2 + 优化器 × 4) / N
            = (2 + 2 + 4) × 参数量 / N
            = 8P / N  bytes

其中 P = 参数量，N = GPU 数量
```

对于 175B 模型，N=8 时：每卡 ≈ 8 × 175B / 8 = 175 GB（比理论略高，有激活值开销）。

## 四、ZeRO-Offload：单卡不够，借 CPU 内存来凑

### 4.1 什么时候需要 Offload？

当 N 再多也凑不齐足够的 GPU 时（比如只有 4 张卡），ZeRO-3 分片后每卡仍然需要 175 GB，单靠 GPU 显存不够。

ZeRO-Offload 的思路：**把 GPU 显存里放不下的东西，临时放到 CPU 内存里**。

```
CPU 内存（DDR4）：通常 256GB - 1TB，单台服务器可以很大
GPU 显存（HBM）：80GB，上限很低
```

### 4.2 两种 Offload 策略

**Offload Optimizer（优化器卸载）**：
- 每张 GPU 的优化器状态（Adam m/v）放到 CPU 内存
- 参数更新时，GPU 从 CPU 拉取状态 → 计算 → 写回
- 适合：优化器状态最大的场景（训练阶段）

**Offload Param（参数卸载）**：
- 更大的模型，参数本身也放不下
- 需要参数时从 CPU 加载到 GPU，用完再换出去
- 适合：超大规模模型推理或训练

**两者可以同时开启**，代价是 CPU-GPU 通信成为瓶颈。

### 4.3 Offload 的性能特点

```
ZeRO-3 + Offload：
优点：可以用很少的 GPU 跑很大的模型（显存突破）
缺点：CPU-GPU 通信（PCIe 64 GB/s）vs NVLink（900 GB/s）差距极大
      速度会显著变慢，通常适合"能跑就行"的场景

ZeRO-3（全 GPU 分片，无 Offload）：
优点：速度快，利用 NVLink/InfiniBand
缺点：需要足够多的 GPU 数量来分片
```

**选择原则**：
- GPU 数量足够（≥8张）→ 纯 ZeRO-3，不用 Offload
- GPU 数量有限（≤4张）→ ZeRO-3 + Offload，能跑但慢
- 极致节省显存 → ZeRO-2/3 + Offload Optimizer

## 五、实战配置：DeepSpeed JSON 配置清单

### 5.1 ZeRO-2 基础配置（推荐起步）

```json
{
  "train_batch_size": 64,
  "gradient_accumulation_steps": 4,
  "fp16": {
    "enabled": true
  },
  "zero_optimization": {
    "stage": 2,
    "offload_optimizer": {
      "device": "none",
      "nproc_per_node": 8
    },
    "overlap_comm": true,
    "contiguous_gradients": true,
    "reduce_bucket_size": 1e6,
    "stage3_prefetch_bucket_size": 1e6,
    "stage3_param_persistence_threshold": 1e5
  },
  "gradient_clipping": 1.0
}
```

**关键字段解释**：

| 字段 | 作用 |
|:---|:---|
| `stage: 2` | 启用 ZeRO-2（优化器状态 + 梯度分片）|
| `overlap_comm: true` | 计算和通信重叠，减少等待时间 |
| `contiguous_gradients: true` | 梯度内存连续分配，提升 AllReduce 效率 |
| `offload_optimizer.device: "none"` | 不卸载到 CPU（全在 GPU）|

### 5.2 ZeRO-3 + Offload（显存受限场景）

```json
{
  "train_batch_size": 32,
  "gradient_accumulation_steps": 8,
  "fp16": {
    "enabled": true
  },
  "zero_optimization": {
    "stage": 3,
    "offload_optimizer": {
      "device": "cpu",
      "pin_memory": true
    },
    "offload_param": {
      "device": "none"
    },
    "overlap_comm": true,
    "contiguous_gradients": true,
    "reduce_bucket_size": 1e6,
    "stage3_max_live_data": 1e8,
    "stage3_max_prefetch_data": 1e8,
    "stage3_param_persistence_threshold": 1e5,
    "stage3_gather_16bit_weights_on_model_save": true
  },
  "gradient_clipping": 1.0
}
```

**关键字段解释**：

| 字段 | 作用 |
|:---|:---|
| `stage: 3` | 启用 ZeRO-3（全部分片）|
| `offload_optimizer.device: "cpu"` | Adam 状态卸载到 CPU 内存 |
| `stage3_gather_16bit_weights_on_model_save: true` | 保存时合并所有分片为完整权重 |
| `pin_memory: true` | 锁页内存，加速 CPU-GPU 传输 |

```json
  },
  "zero_optimization": {
    "stage": 3,
    "offload_optimizer": {
      "device": "none"
    },
    "offload_param": {
      "device": "none"
    },
    "overlap_comm": true,
    "contiguous_gradients": true,
    "reduce_bucket_size": 1e8,
    "stage3_prefetch_bucket_size": 1e8,

```

**与 Offload 配置的核心区别**：所有 `device` 设为 `"none"`，全部状态留在 GPU，分片靠足够的 GPU 数量支撑。

## 六、ZeRO 与其他并行策略的关系

ZeRO 是一个**正交维度**，可以和其他并行策略自由组合：

```
ZeRO-3（显存分片）
    +
Pipeline Parallel（层间切分，减少每卡层数）
    +
Tensor Parallel（单节点内 NVLink，加速分片内通信）
    +
Data Parallel（增加并发训练的 batch 数）
```

这就是 DeepSpeed 官方推荐的 **3D 并行 + ZeRO** 方案。

**实际配置举例（训练 70B 模型，8节点 × 8卡 A100）**：
- 总 GPU：64 张
- TP = 8（节点内张量并行）
- PP = 8（流水线并行，8个阶段）
- DP = 8（数据并行，8路）
- ZeRO = Stage 3（每卡只存 1/8 的状态）

```
每卡显存估算：
= (模型参数 140GB) / (TP × PP) / DP_ZeRO_sharding
= 140GB / 8 / 8 / 8
≈ 0.27 GB（主要放激活值和临时计算）
→ 完全可行
```

## 七、核心知识点

1. **显存瓶颈根源**：训练显存 ≈ 参数 × 4（梯度+Adam）+ 激活值，远超单卡上限
2. **DP vs 模型并行**：DP 不省显存只是省时间；模型并行省显存但增加通信复杂度
3. **ZeRO 三个 Stage**：本质是把每张卡的"完整副本"变成"1/N 分片"，N 越大显存越省
4. **显存节省公式**：ZeRO-3 每卡 ≈ 8P/N bytes（P=参数量，N=GPU数）
5. **Offload 权衡**：用 CPU 内存换显存，速度变慢，适合 GPU 数量不足的场景
6. **实战选择**：GPU 够多（≥8卡/节点）→ ZeRO-3；GPU 少 → ZeRO-3 + Offload
