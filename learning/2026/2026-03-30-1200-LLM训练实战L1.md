## 一、为什么单个 GPU 装不下 LLM？

### 1.1 参数量的物理含义

GPT-3 175B（Billion）参数：
- 每个参数在 FP16（半精度浮点数）下占 **2 bytes**
- 175B × 2 bytes = **350 GB**
- 最强单卡 A100/H100 = 80GB HBM

**缺口：350GB vs 80GB = 差了 4.4 倍**

这意味着没有任何单卡能放下 GPT-3 175B 的完整模型。必须把模型拆分到多张卡上。

### 1.2 训练 vs 推理的显存占用

| 阶段 | 显存占用项 | 典型大小（175B模型） |
|:---|:---|:---|
| **推理** | 模型参数（FP16） | 350GB |
| | KV Cache（上下文越长越大） | 可达数十 GB |
| **训练** | 模型参数 | 350GB |
| | 梯度（与参数量相同） | 350GB |
| | 优化器状态（Adam：momentum + variance）| 700GB（2个FP32副本） |
| | 激活值（BatchSize×SequenceLen×层数）| 数十 GB |
| **总计** | 推理 | ~350GB + KV Cache |
| | 训练 | ~1400GB（需要14张80GB卡） |

这就是为什么训练 LLM 需要"超节点"——几百张 GPU 组成的集群。

## 二、三种并行策略的本质

分布式训练有三种基本并行方式，现实中是它们的组合。

### 2.1 数据并行（Data Parallelism, DP）

**核心思想**：每张卡都有模型的完整副本，处理不同的数据批次。

```
场景：8张 GPU，BatchSize=128

GPU 0: model_copy, batch[0:16]    → 计算 loss
GPU 1: model_copy, batch[16:32]  → 计算 loss
GPU 2: model_copy, batch[32:48]  → 计算 loss
...（7张卡并行）
GPU 7: model_copy, batch[112:128] → 计算 loss

→ 8张卡各自得到梯度
→ AllReduce：所有卡的梯度求平均
→ 每张卡用相同的梯度更新自己的模型副本
→ 8张卡的模型保持同步（相同）
```

**优点**：
- 实现最简单
- 扩展性好：N 张卡 → 约 N 倍吞吐
- 每张卡有完整模型，显存要求和单卡相同

**缺点**：
- **不节省显存**：每张卡都要装下完整模型
- 适合：模型能装进单卡，但数据量大、训练慢

**PyTorch 实现**：
```python
model = DistributedDataParallel(model, device_ids=[local_rank])
# 框架自动处理：分割数据 → 各卡前向 → AllReduce 梯度 → 各卡更新
```

### 2.2 张量并行（Tensor Parallelism, TP）

**核心思想**：把**一个层**的参数矩阵横向拆分，分布到多张卡上。

**举例：矩阵乘法 Y = X · W**

```
原始（单卡）：
X [seq, 4096] @ W [4096, 14336] → Y [seq, 14336]

张量并行（4卡，列切）：
X [seq, 4096] @ W₀ [4096, 3584] → Y₀ [seq, 3584]   → GPU 0
X [seq, 4096] @ W₁ [4096, 3584] → Y₁ [seq, 3584]   → GPU 1
X [seq, 4096] @ W₂ [4096, 3584] → Y₂ [seq, 3584]   → GPU 2
X [seq, 4096] @ W₃ [4096, 3584] → Y₃ [seq, 3584]   → GPU 3

→ 4张卡的输出 Y₀...Y₃ 拼起来 = Y [seq, 14336]
```

**关键技术：AllGather 通信**
- 做完这个线性层后，需要一次 AllGather 把所有分片拼起来
- 通信量 = O(seq × d_model)，跨卡带宽要求极高

**优点**：节省显存（单卡只存 1/N 的参数）
**缺点**：通信极频繁，必须有 NVLink（卡间高速互联）才能高效

**适用场景**：超节点内部，GPU 通过 NVLink 直连

**Megatron-LM** 是最成熟的张量并行框架。

### 2.3 流水线并行（Pipeline Parallelism, PP）

**核心思想**：把**不同的层**放到不同 GPU 上。

```
场景：40层模型，4张GPU

GPU 0: 层 0-9     （最前面）
GPU 1: 层 10-19
GPU 2: 层 20-29
GPU 3: 层 30-39   （最后面）

输入: "Hello world"
→ GPU 0 处理层 0-9  → 中间结果 h₁₀
→ h₁₀ 传给 GPU 1   （通信：中间激活）
→ GPU 1 处理层 10-19 → h₂₀
→ ...以此类推
```

**问题：GPU 空闲（流水线气泡）**

朴素实现下，同一时间只有一张卡在工作，其他卡在等待：

```
时间 →  t1   t2   t3   t4   t5   t6   t7   t8
GPU0 [F0 ][F0 ][F0 ][W0 ][F0 ][F0 ][F0 ][W0 ]  ← 一直在工作
GPU1 [idle][F1 ][F1 ][F1 ][W1 ][F1 ][F1 ][F1 ]  ← 等GPU0
GPU2 [idle][idle][F2 ][F2 ][F2 ][W2 ][F2 ][F2 ] ← 等GPU1
GPU3 [idle][idle][idle][F3 ][F3 ][F3 ][W3 ][W3 ]← 等GPU2
```

**解决方案：Micro-Batch 流水线**
- 把一个 Batch 切成多个 Micro-Batch
- 让不同 GPU 可以同时处理不同 Micro-Batch
- 大幅减少气泡，但无法完全消除

**通信量**：只有层间的中间激活（较少），适合跨节点（网络带宽有限）

### 2.4 三种并行对比

| 维度 | 数据并行 | 张量并行 | 流水线并行 |
|:---|:---|:---|:---|
| **模型切分方式** | 每卡完整副本 | 单层内横向切 | 不同层放不同卡 |
| **显存节省** | ❌ 不节省 | ✅ 显著节省 | ✅ 显著节省 |
| **通信模式** | AllReduce（梯度）| AllGather（激活）| P2P（激活传一次）|
| **通信量** | 中等 | 极大 | 小 |
| **通信频率** | 每反向传播一次 | 每个 Transformer 层多次 | 每层一次 |
| **对硬件要求** | 普通网络即可 | 必须 NVLink | 跨节点网络 |
| **扩展效率** | 接近线性 | 受通信瓶颈限制 | 有流水线气泡 |

## 三、实战：DeepSpeed ZeRO 分布式策略

DeepSpeed 是微软的分布式训练框架，核心是 **ZeRO（Zero Redundancy Optimizer）**。

### 3.1 ZeRO 的三个 Stage

**ZeRO-Stage 1：分片优化器状态**
```
普通DDP：每张卡存完整的 [Adam m, Adam v] → 优化器状态 700GB/卡
ZeRO-1：  每张卡只存 1/N 的优化器状态
→ 显存节省约 1/4（N=4时）
```

**ZeRO-Stage 2：分片优化器状态 + 梯度**
```
ZeRO-2：每张卡存 1/N 的优化器状态 + 1/N 的梯度
→ 显存节省约 1/2（N=4时）
```

**ZeRO-Stage 3：分片所有状态（参数+梯度+优化器）**
```
ZeRO-3：每张卡只存 1/N 的完整模型状态
→ 显存节省理论上 N 倍（但通信量最大）
```

### 3.2 ZeRO 通信量分析

ZeRO 的代价是通信：每次反向传播后，需要 AllReduce 来 gather 梯度/参数。

```
ZeRO-1: 通信量 ≈ 2×参数大小（gradient allreduce + 参数 scatter）
ZeRO-2: 通信量 ≈ 2×参数大小 + 梯度
ZeRO-3: 通信量 ≈ 3×参数大小
```

但通信量和显存节省的 trade-off 非常值得。对于大模型，ZeRO-3 是标配。

### 3.3 DeepSpeed 配置示例

```json
{
  "train_batch_size": 128,
  "gradient_accumulation_steps": 4,
  "fp16": {"enabled": true},
  "zero_optimization": {
    "stage": 3,
    "offload_optimizer": {"device": "cpu"},
    "offload_param": {"device": "cpu"},
    "overlap_comm": true,
    "contiguous_gradients": true
  },
  "gradient_clipping": 1.0
}
```

**offload_param/offload_optimizer**：把不常用的参数/优化器状态卸载到 CPU 内存，突破 GPU 显存限制。

## 四、混合并行实战：3D 并行

真实大模型训练（如 LLaMA、GPT-4 级别）用的是 **3D 并行**：

```
数据并行 (DP)  ─── 增大 Batch Size
    ↓
流水线并行 (PP) ─── 跨 GPU 节点，层间切分
    ↓
张量并行 (TP)  ─── 单节点内部，层内切分（需要 NVLink）
```

**典型配置（训练 175B 模型）**：
- 8个节点，每个节点 8张 A100（ NVLink）
- TP=8（每节点内 8卡张量并行）
- PP=8（8个阶段流水线）
- DP=64（8节点 × 8卡数据并行）
- 总 GPU 数 = 8 × 8 = 64 张

**Megatron-DeepSpeed 组合**：用 Megatron 做 TP/PP，用 DeepSpeed 做 DP 和 ZeRO。

## 五、NVIDIA 集群的硬件层次

理解硬件层次才能理解并行策略的选择：

```
一台服务器（节点）
├── 8 × GPU（A100/H100）
│   ├── 每卡 80GB HBM（高带宽显存）
│   └── NVLink：卡间互联（900 GB/s 双向）
├── 2 × CPU
│   └── DDR4/PCIe：CPU-GPU 通信（64 GB/s）
└── InfiniBand：节点间互联（400-800 Gbps）
```

**选择并行策略的关键**：
- **NVLink 可用** → 可以用 TP，带宽充足
- **只有 InfiniBand** → 用 PP+DP，避免高频率跨卡通信
- **CPU 内存大** → 用 ZeRO-offload，把状态卸到 CPU

## 六、常见误区

| 误区 | 正确理解 |
|:---|:---|
| "数据并行节省显存" | 错！每卡都有完整模型副本，显存一样多 |
| "张量并行适合跨节点" | 错！TP 通信极频繁，需要 NVLink，不适合跨节点 |
| "流水线并行没有通信" | 错！有通信，但只有层间激活一次，量不大 |
| "ZeRO-3 最省显存就用它" | 通信量也最大，需要高带宽网络 |
| "A100 80GB 能跑 175B 训练" | 不能！训练需要 1400GB+，必须分布式 |

## 七、核心知识点

1. **训练 LLM 的显存消耗** = 参数（FP16）× 4~8 倍（梯度+优化器+激活）
2. **数据并行**：不省显存，省时间（N 卡 ≈ N 倍速）
3. **张量并行**：省显存，但需要 NVLink，适合节点内
4. **流水线并行**：省显存，适合节点间，但有气泡
5. **ZeRO**：用通信换显存，三阶段逐步加大节省力度
6. **3D 并行**：真实大模型训练的标准配置
