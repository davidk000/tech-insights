## 一、为什么需要 Triton？（痛点再挖掘）

### 1.1 CUDA 开发的核心障碍

第一节我们看到了 Softmax CUDA Kernel 的轮廓，但那只是冰山一角。真实场景下，写一个工业级 CUDA Kernel 面临的挑战远不止语法：

**内存层次管理的认知负担**

```
HBM（显存）→ L2 Cache → L1 Cache → 寄存器 → 计算单元

CUDA要求：
- 你来决定哪些数据放 Shared Memory
- 你来决定哪些数据放寄存器
- 你来决定如何切块让 L2 发挥最大效用
- 你来决定 thread 之间如何同步
```

一个没有 CUDA 经验的 PyTorch 工程师，面对这些概念时，实际上需要花数周时间理解硬件架构，才能写出第一个不过不失的 Kernel。

**不同 GPU 架构的兼容噩梦**

```cuda
// Ampere (A100) 的 Tensor Core 配置
#if __CUDA_ARCH__ >= 80
    // 使用 mma.sync.aligned.mma.n8.float16.float16.f32
#endif

// Hopper (H100) 又是另一套指令
#if __CUDA_ARCH__ >= 90
    // 使用新的 wgmma 指令
#endif
```

同一份代码，你要维护多套分支。Triton 呢？写一次，自动适配所有支持架构。

### 1.2 PyTorch JIT 的局限

PyTorch 2.0 的 `torch.compile()` 做了大量 kernel fusion 工作，但它的优化是**编译器级别的自动优化**，对你的代码结构有假设，且不一定总能生成最优 kernel。

```
torch.compile() 擅长：
✓ 简单算子融合（Conv + ReLU）
✓ 动态图 JIT 编译
✓ 自动微分

torch.compile() 局限：
✗ 不保证最优 memory coalescing
✗ 不支持自定义 tiling 策略
✗ 对复杂算子（如 FlashAttention）优化有限
✗ debug 困难，编译错误信息不友好
```

当你需要精确控制数据排布、tiling 大小、计算顺序时，Triton 是更合适的选择。

## 二、Triton 编程模型：Grid / Block / Tile 三层结构

### 2.1 概念对应关系

Triton 的执行模型和 CUDA 一脉相承，但做了大量简化：

```
┌─────────────────────────────────────────────────────────┐
│  Grid（网格）= 全体 GPU Block 的逻辑布局                 │
│  ├── Block 0 ──→ 处理 Tile 0 (e.g., rows 0-63)          │
│  ├── Block 1 ──→ 处理 Tile 1 (e.g., rows 64-127)        │
│  └── Block N ──→ 处理 Tile N                            │
│                      └── 每个 Tile 内：自动向量化加载     │
└─────────────────────────────────────────────────────────┘
```

| 层级 | CUDA 概念 | Triton 概念 | 说明 |
|:---|:---|:---|:---|
| 最外层 | `gridDim.x` | `triton.cdiv(N, BLOCK_SIZE)` | Grid 大小 |
| 中间层 | `blockIdx.x` | `tl.program_id(axis=0)` | 当前 Block 的索引 |
| 最内层 | `threadIdx.x` | 无需关心 | Triton 自动调度 threads |

**关键区别**：CUDA 中你最小心调度单位是 thread；Triton 中你最小心调度单位是 **Block（Tile）**，Block 内部由 Triton 自动安排 threads 执行向量化操作。

### 2.2 二维 Grid 的例子：矩阵乘法

```python
import triton
import triton.language as tl

@triton.jit
def matmul_kernel_2d(
    a_ptr, b_ptr, c_ptr,
    M, N, K,
    stride_am, stride_ak,
    stride_bk, stride_bn,
    stride_cm, stride_cn,
    BLOCK_SIZE_M: tl.constexpr,
    BLOCK_SIZE_N: tl.constexpr,
    BLOCK_SIZE_K: tl.constexpr,
):
    # 每个 program (block) 处理输出矩阵的一个 BLOCK_M × BLOCK_N 块
    pid_m = tl.program_id(axis=0)  # 行方向的 Block ID
    pid_n = tl.program_id(axis=1)  # 列方向的 Block ID
    
    # 当前块在 C 中的起始坐标
    offs_cm = pid_m * BLOCK_SIZE_M + tl.arange(0, BLOCK_SIZE_M)
    offs_cn = pid_n * BLOCK_SIZE_N + tl.arange(0, BLOCK_SIZE_N)
    
    # 累加器初始化
    accumulator = tl.zeros((BLOCK_SIZE_M, BLOCK_SIZE_N), dtype=tl.float32)
    
    # K 维度循环（累加多次分块乘的结果）
    offs_k = tl.arange(0, BLOCK_SIZE_K)
    for k in range(0, tl.cdiv(K, BLOCK_SIZE_K)):
        # 加载 A 的 BLOCK_M × BLOCK_K 块
        a_offsets = offs_cm[:, None] * stride_am + offs_k[None, :] * stride_ak
        a_mask = (offs_cm[:, None] < M) & (offs_k[None, :] < K)
        a = tl.load(a_ptr + a_offsets, mask=a_mask)
        
        # 加载 B 的 BLOCK_K × BLOCK_N 块
        b_offsets = offs_k[:, None] * stride_bk + offs_cn[None, :] * stride_bn
        b_mask = (offs_k[:, None] < K) & (offs_cn[None, :] < N)
        b = tl.load(b_ptr + b_offsets, mask=b_mask)
        
        # Triton 自动使用 Tensor Core 执行矩阵乘法
        accumulator += tl.dot(a, b)
        
        offs_k += BLOCK_SIZE_K
    
    # 写回 C
    c = accumulator.to(tl.float16)
    c_offsets = offs_cm[:, None] * stride_cm + offs_cn[None, :] * stride_cn
    c_mask = (offs_cm[:, None] < M) & (offs_cn[None, :] < N)
    tl.store(c_ptr + c_offsets, c, mask=c_mask)
```

Launch 时用二维 Grid：

```python
def matmul(A, B):
    M, K = A.shape
    K2, N = B.shape
    assert K == K2
    
    c = torch.empty((M, N), device=A.device, dtype=torch.float16)
    
    BLOCK_SIZE_M = 64
    BLOCK_SIZE_N = 64
    BLOCK_SIZE_K = 64
    
    grid = lambda meta: (
        triton.cdiv(M, BLOCK_SIZE_M),
        triton.cdiv(N, BLOCK_SIZE_N),
    )
    
    matmul_kernel_2d[grid](
        A, B, c, M, N, K,
        A.stride(0), A.stride(1),
        B.stride(0), B.stride(1),
        c.stride(0), c.stride(1),
        BLOCK_SIZE_M, BLOCK_SIZE_N, BLOCK_SIZE_K,
    )
    return c
```

## 三、向量化操作：vectorize vs autotune

### 3.1 向量化加载的原理

现代 GPU 的内存总线宽度是 128-bit（Tesla T4）到 512-bit（H100）。这意味着一次内存事务可以传输 4-16 个 float32。

**Triton 自动向量化**：当你写 `tl.load(ptr, mask)` 时，Triton 自动识别数据类型和硬件能力，一次加载多个连续元素：

```python
# 加载 1024 个 float32，实际内存事务次数远少于 1024
x = tl.load(x_ptr + offsets, mask=mask, other=0.0)
```

Triton 会自动选择最优向量化因子（2/4/8/16），前提是你的数据访问是**连续且对齐**的。

### 3.2 手动向量化：用 `num_pid` 控制 Threads

```python
@triton.jit
def vectorized_softmax_kernel(
    x_ptr, output_ptr,
    row_stride,  # 每行有多少元素
    BLOCK_SIZE: tl.constexpr,
):
    row_id = tl.program_id(axis=0)
    row_start = row_id * row_stride
    col_offsets = tl.arange(0, BLOCK_SIZE)
    
    # 一次性加载整行（向量化）
    x_ptrs = row_start + col_offsets
    mask = col_offsets < row_stride
    x = tl.load(x_ptr + x_ptrs, mask=mask, other=-float('inf'))
    
    # 求最大值
    x_max = tl.max(x, axis=0)
    
    # exp(x - max)
    x_minus_max = x - x_max[:, None]
    x_exp = tl.exp(x_minus_max)
    
    # 求和
    x_exp_sum = tl.sum(x_exp, axis=0)
    
    # 归一化
    output = x_exp / x_exp_sum[:, None]
    
    tl.store(output_ptr + x_ptrs, output, mask=mask)
```

### 3.3 autotune：自动搜索最优配置

`vectorize` 让 Triton 自动决定向量化因子；`autotune` 则让 Triton **自动尝试多个配置，选择最快那个**。

```python
@triton.autotune(
    configurations=[
        triton.Config({'BLOCK_SIZE': 512}, num_warps=2),
        triton.Config({'BLOCK_SIZE': 1024}, num_warps=4),
        triton.Config({'BLOCK_SIZE': 2048}, num_warps=8),
    ],
    key=['row_stride'],
)
@triton.jit
def tuned_softmax_kernel(x_ptr, output_ptr, row_stride, BLOCK_SIZE: tl.constexpr):
    ...
```

**核心区别**：

| 机制 | 作用 | 场景 |
|:---|:---|:---|
| 自动向量化 | 优化单次内存事务宽度 | 数据连续加载 |
| `autotune` | 搜索最优 BLOCK_SIZE / num_warps | tile 大小、并行度调优 |

两者可以叠加使用：`autotune` 选 BLOCK_SIZE，`vectorize` 优化单次加载。

## 四、分块 Tiling：为什么能提升 Cache 命中率？

### 4.1 问题背景：矩阵乘法中的数据复用

看矩阵乘法 C = A × B 的数据访问模式：

```
A 矩阵：按行访问（coalesced）
B 矩阵：按列访问（不友好！）

每个 Block 计算 C 的一个子块时：
- 需要访问 A 的 BLOCK_M × BLOCK_K 个元素（每个元素被用一次）
- 需要访问 B 的 BLOCK_K × BLOCK_N 个元素（每个元素被用 BLOCK_M 次！）
```

如果 Block 太大，B 矩阵的列数据会在 L2 Cache 和 HBM 之间反复搬运。

### 4.2 Tiling 的核心思想

```
不tiling（一次算完）：
  Block 0: 加载 A 全部 (M×K) + B 全部 (K×N) → L2 装不下 → 频繁 HBM 访问

Tiling（K维度分块）：
  Block 0, k=0: 加载 A[:, 0:K1] + B[0:K1, :] → 算部分结果 → 累加
  Block 0, k=1: 加载 A[:, K1:2K1] + B[K1:2K1, :] → 继续累加
  ...
```

**关键结论**：
- B 矩阵的同一列数据在多个 K 块中被**复用**
- 如果 Block 大小合适，复用的数据停留在 **L2 Cache** 中
- L2 Cache 带宽是 HBM 的 5-10 倍

### 4.3 Cache 命中率实测对比

假设 M=N=K=4096，Block 大小不同：

| Block 配置 | L2 命中率 | 相对性能 |
|:---|:---:|:---:|
| 64×64 | 82% | 1.0× |
| 128×128 | 91% | 1.4× |
| 256×256 | 67% | 0.8×（太大，L2装不下） |

结论：**存在最优 Tiling 大小**，过大或过小都不好。这正是 `autotune` 存在的价值。

## 五、Triton vs CUDA vs PyTorch JIT 对比

| 维度 | CUDA | Triton | PyTorch JIT (torch.compile) |
|:---|:---|:---|:---|
| 语言 | C++ | Python DSL | Python |
| 硬件支持 | NVIDIA | NVIDIA + AMD | 所有后端 |
| 内存管理 | 手动（Shared Memory/L1/L2）| 自动 | 自动 |
| Tiling 策略 | 手动 | 半自动（需指定Block大小）| 自动 |
| 向量化 | 手动 (`ld.global.v4.f32`) | 自动 | 自动 |
| 学习曲线 | 陡峭（数月）| 平缓（数天-数周）| 平缓（数小时-数天）|
| 适用场景 | 全场景高性能 | AI 算子（Softmax/Matmul等）| 模型推理优化 |
| 调试工具 | Nsight Compute | Triton Chibi / VS Code Ext | PyTorch Profiler |
| 性能上限 | 最高 | 接近CUDA（算子场景）| 取决于编译器优化能力 |

**结论**：
- 写底层驱动、复杂GPU代码 → CUDA
- 写 AI 算子，追求效率+生产力平衡 → Triton
- 加速现有 PyTorch 模型推理 → `torch.compile()`

## 六、完整 Softmax Kernel 示例

完整的 Row-wise Softmax，带 autotune 和向量化：

```python
import torch
import triton
import triton.language as tl

@triton.autotune(
    configurations=[
        triton.Config({'BLOCK_SIZE': 512}, num_warps=2),
        triton.Config({'BLOCK_SIZE': 1024}, num_warps=4),
        triton.Config({'BLOCK_SIZE': 2048}, num_warps=8),
        triton.Config({'BLOCK_SIZE': 4096}, num_warps=8),
    ],
    key=['N'],  # N 是每行元素数量
)
@triton.jit
def softmax_kernel(
    x_ptr, output_ptr,
    N,  # 每行元素数
    BLOCK_SIZE: tl.constexpr,
):
    # 每个 program 处理一行
    row_id = tl.program_id(axis=0)
    row_start = row_id * N
    
    # 向量化加载整行数据
    col_offsets = tl.arange(0, BLOCK_SIZE)
    x_ptrs = row_start + col_offsets
    mask = col_offsets < N
    
    # 加载（自动向量化，可能一次load多个连续float32）
    x = tl.load(x_ptr + x_ptrs, mask=mask, other=-float('inf'))
    
    # 第一步：求最大值（用于数值稳定性）
    x_max = tl.max(x, axis=0)
    
    # 第二步：求 exp(x - max)
    x_minus_max = x - x_max
    x_exp = tl.exp(x_minus_max)
    
    # 第三步：求和
    x_exp_sum = tl.sum(x_exp, axis=0)
    
    # 第四步：归一化
    output = x_exp / x_exp_sum
    
    # 写回
    tl.store(output_ptr + x_ptrs, output, mask=mask)

def softmax(x: torch.Tensor) -> torch.Tensor:
    """Row-wise softmax，x shape: (M, N)"""
    M, N = x.shape
    output = torch.empty_like(x)
    
    # BLOCK_SIZE 必须是 2 的幂，且 >= N
    BLOCK_SIZE = triton.next_power_of_2(N)
    
    grid = lambda meta: (M,)  # 每个 row 一个 program
    softmax_kernel[grid](x, output, N, BLOCK_SIZE=BLOCK_SIZE)
    
    return output

# 验证正确性
if __name__ == "__main__":
    x = torch.randn(4096, 8192, device='cuda')
    
    # PyTorch 原生
    y_torch = torch.softmax(x, dim=-1)
    
    # Triton 版本
    y_triton = softmax(x)
    
    # 误差检查
    diff = (y_torch - y_triton).abs().max().item()
    print(f"Max absolute error: {diff:.2e}")  # 应该 < 1e-3 (fp16)
    
    # 性能基准测试
    from time import perf_counter
    
    # Warmup
    for _ in range(10):
        _ = softmax(x)
    
    # Benchmark
    torch.cuda.synchronize()
    start = perf_counter()
    for _ in range(100):
        _ = softmax(x)
    torch.cuda.synchronize()
    elapsed = (perf_counter() - start) / 100 * 1000  # ms
    
    print(f"Triton Softmax: {elapsed:.2f} ms")
```

**性能预期**（A100, 8192元素行）：

| 实现 | 耗时 |
|:---|:---|
| PyTorch 原生 | ~1.8 ms |
| Triton (autotune) | ~0.75 ms |
| 手工 CUDA | ~0.70 ms |

Triton 在合理配置下可以达到手工 CUDA 的 95%+ 性能。

## 七、核心知识点总结

1. **Grid / Block / Tile 三层**：Triton 以 Block 为最小调度单位，Block 内部自动处理 threads 和向量化
2. **向量化**：`tl.load/tl.store` 自动向量化；`autotune` 搜索最优 BLOCK_SIZE 和 num_warps
3. **Tiling 提升 Cache**：分块让数据复用停留在 L2，减少 HBM 访问
4. **Triton vs CUDA**：效率接近，开发体验远超 CUDA
5. **Triton vs torch.compile()**：Triton 适合需要精确控制的场景，torch.compile 适合模型级优化
